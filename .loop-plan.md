## Tasks

### Task 1: Count-mismatch → partial-result handling (src/categorize.rs:414-452)

**Problem:** When the LLM returns 3 records for 5 segments, `categorize_rich`
discards ALL 3 good records and falls back to rules for the entire chunk →
empty tags, summary = user message. This is the #1 quality gap.

**Fix:** Use the parsed records that DID succeed (up to `min(parsed.len(),
chunk.len())`), fill the remaining slots with rules-fallback records.

**Refactor:** Extract the duplicated rules-fallback closure (lines 428-440 and
442-451 are identical) into a private helper `rules_fallback_record(&self, seg)
-> RichRecord` to eliminate the copy-paste.

**MIRROR:** src/categorize.rs:414-452 (categorize_rich structure),
src/categorize.rs:349-407 (parse_rich — returns Vec<RichRecord>).

**Verification:**

- New test `count_mismatch_uses_partial_results`: mock returns 3 records for 5
  segments → result has 3 LLM records (with tags+summary) + 2 rules-filled
  (empty tags, summary=user_message). Assert: `result[0].tags == ["auth"]`,
  `result[3].tags.is_empty()`, `result.len() == 5`.
- New test `llm_error_falls_back_to_rules_for_chunk`: mock returns `Err("timeout")`
  → all 5 records are rules-filled. Assert: `result.len() == 5`,
  `result[0].tags.is_empty()`.
- All 8 existing categorize tests still pass (the mock that returns exact-count
  records still works — partial path only triggers on mismatch).
